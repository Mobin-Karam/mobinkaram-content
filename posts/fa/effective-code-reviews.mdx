بازبینی خوب ریسک را زود می‌گیرد بدون اینکه تیم را کند کند. این راهنمای کوتاه را می‌توانید از فردا در گردش کار خود استفاده کنید.

## هدف‌های بازبین

- **درستی و ریسک:** باگ، فساد داده، خلا امنیتی.  
- **هم‌ترازی:** مرزبندی معماری، قراردادها و الزامات محصول.  
- **نگهداشت‌پذیری:** شفافیت، تست‌ها و قلاب‌های observability.

## چک‌لیست بازبین

- PR کوچک نگه دارید (<۴۰۰ خط تغییر) یا درخواست split کنید.  
- اول **قصد** را بخوانید (issue، design doc، توضیح PR).  
- قبل از جزئیات، **رابط‌ها و قراردادها** را بررسی کنید.  
- دنبال **تست‌های جاافتاده** بگردید: edge case، الگوهای flaky، نبود تست رگرسیون برای باگ.  
- **Observability** را چک کنید: لاگ/متریک/تریس برای جریان جدید یا مسیر خطا.

## عادت‌های نویسنده که سرعت می‌دهد

- توضیح PR شفاف: مسئله، رویکرد، trade-off، تست‌ها.  
- برای تغییرات UI **اسکرین‌شات/کلیپ** بگذارید؛ برای کارایی به **متریک** لینک دهید.  
- جاهایی که نیاز به دقت بیشتر دارید علامت بزنید.  
- کامیت‌ها را منطقی و با تست‌های پاس نگه دارید.

## تصمیم‌گیری

- **کامنت blocking** فقط برای درستی/ایمنی؛ برای سبک non-blocking بگذارید.  
- در اختلاف، **گزینه جایگزین** پیشنهاد دهید یا داده بخواهید.  
- نتیجه را در خلاصه PR یا ADR ثبت کنید اگر معماری را لمس می‌کند.

## متریک‌هایی که باید دید

- زمان تا اولین پاسخ.  
- عمق بازبینی: تعداد کامنت‌های جدی نسبت به نیت‌ها.  
- باگ‌های بعد از مرج.  
- توزیع اندازه PR.

## ضدالگوها

- LGTM سرسری.  
- کمال‌گرایی بدون توجه به بیزنس.  
- انبوه نیت بدون اتوماسیون فرمت/لینت.  
- مالکیت مبهم: چه کسی approve می‌کند و مسئول ریسک است؟

## تمپلیت سبک (قابل کپی)

```
## Context
## Changes
## Risks
## Tests
## Screenshots/metrics
## Notes for reviewers
```

بازبینی کد یک مهارت است. با عادت‌های آگاهانه و اندازه‌گیری، سریع‌تر و با غافلگیری کمتر ship می‌کنید.
